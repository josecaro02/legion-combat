## Arquitectura

### Estructura de archivos

```
legion-combat/
├── alembic.ini              # Configuración principal de Alembic
├── migrations/
│   ├── env.py               # Script de configuración del entorno
│   ├── script.py.mako       # Template para nuevas migraciones
│   ├── README               # Documentación interna
│   └── versions/            # Scripts de migración (*.py)
│       ├── 3fda168627d5_initial_schema.py
│       └── ...
└── app/
    ├── models/              # Modelos SQLAlchemy
    │   ├── __init__.py      # Importa todos los modelos
    │   ├── base.py          # TimestampMixin
    │   ├── user.py
    │   ├── student.py
    │   ├── payment.py
    │   ├── schedule.py
    │   ├── class_instance.py
    │   └── attendance.py
    └── extensions.py        # Base y db
```

### Cómo funciona la autogeneración

1. Alembic lee `migrations/env.py`
2. `env.py` importa `Base` desde `app.extensions`
3. `env.py` importa todos los modelos desde `app.models`
4. Alembic compara `Base.metadata` con el esquema actual de la BD
5. Genera un script de migración con las diferencias

## Configuración

### Variables de Entorno

Alembic usa `DATABASE_URL` desde el entorno:

```bash
# .env
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/legion_combat_dev
```

Si no está definida, usa un valor por defecto de desarrollo.

### Configuración de env.py

El archivo `migrations/env.py` está configurado para:

- Importar automáticamente todos los modelos desde `app.models`
- Usar `Base.metadata` para la autogeneración
- Soportar SQLAlchemy 2.0 con `render_as_batch=True`
- Comparar tipos de columnas con `compare_type=True`

## Comandos Principales

### Información

```bash
# Ver versión actual aplicada en la BD
alembic current

# Ver historial completo de migraciones
alembic history

# Ver historial con más detalle
alembic history --verbose
```

### Crear migraciones

```bash
# Crear migración autogenerada (recomendado)
alembic revision --autogenerate -m "descripcion del cambio"

# Crear migración vacía (para escribir manualmente)
alembic revision -m "nombre de la migracion"
```

### Aplicar migraciones

```bash
# Aplicar todas las migraciones pendientes
alembic upgrade head

# Aplicar hasta una versión específica
alembic upgrade 3fda168627d5

# Aplicar solo la siguiente migración
alembic upgrade +1
```

### Revertir migraciones

```bash
# Revertir la última migración
alembic downgrade -1

# Revertir hasta una versión específica
alembic downgrade 3fda168627d5

# Revertir todo (volver al inicio)
alembic downgrade base
```

## Flujo de Trabajo

### Desarrollo local

#### 1. Modificar un modelo

```python
# app/models/student.py

class Student(Base, TimestampMixin):
    __tablename__ = "students"
    
    # ... campos existentes ...
    
    # Nuevo campo
    emergency_contact: Mapped[Optional[str]] = mapped_column(
        String(100),
        nullable=True
    )
```

#### 2. Crear la migración

```bash
alembic revision --autogenerate -m "agregar contacto de emergencia"
```

#### 3. Revisar el script generado

```bash
# El archivo estará en migrations/versions/
cat migrations/versions/xxxx_agregar_contacto_de_emergencia.py
```

Ejemplo de script generado:

```python
def upgrade() -> None:
    # ### commands auto generated by Alembic ###
    op.add_column('students', sa.Column('emergency_contact', sa.String(length=100), nullable=True))
    # ### end Alembic commands ###

def downgrade() -> None:
    # ### commands auto generated by Alembic ###
    op.drop_column('students', 'emergency_contact')
    # ### end Alembic commands ###
```

#### 4. Aplicar la migración

```bash
alembic upgrade head
```

#### 5. Verificar en la base de datos

```bash
psql -U postgres -d legion_combat_dev -c "\d students"
```

## Escenarios Comunes

### Agregar una nueva columna

```python
# En el modelo
new_column: Mapped[Optional[str]] = mapped_column(String(50), nullable=True)

# Migración generada (solo nueva columna)
op.add_column('table_name', sa.Column('new_column', sa.String(length=50), nullable=True))
```

### Cambiar tipo de columna

```python
# Antes
age: Mapped[int] = mapped_column(Integer, nullable=False)

# Después
age: Mapped[float] = mapped_column(Float, nullable=False)

# Migración generada
op.alter_column('table_name', 'age',
    existing_type=sa.INTEGER(),
    type_=sa.Float(),
    existing_nullable=False
)
```

### Crear un nuevo índice

```python
# En el modelo
__table_args__ = (
    Index('ix_custom', 'field1', 'field2'),
)

# Migración generada
op.create_index('ix_custom', 'table_name', ['field1', 'field2'], unique=False)
```

### Relación foreign key

```python
# En el modelo
other_id: Mapped[uuid.UUID] = mapped_column(ForeignKey("others.id"), nullable=False)

# Migración generada
op.create_foreign_key(None, 'table_name', 'others', ['other_id'], ['id'])
```

### Crear tabla nueva

```bash
# 1. Crear el modelo en app/models/new_model.py
# 2. Importar en app/models/__init__.py
# 3. Generar migración
alembic revision --autogenerate -m "crear tabla new_model"
# 4. Aplicar
alembic upgrade head
```

## Resolución de Problemas

### Alembic no detecta mis cambios

**Causas comunes:**

1. **El modelo no está importado en `app/models/__init__.py`**
   ```python
   from app.models.new_model import NewModel
   ```

2. **El modelo no hereda de `Base`**
   ```python
   from app.extensions import Base  # Correcto
   # NO from sqlalchemy.orm import DeclarativeBase
   ```

3. **Sintaxis incorrecta en el modelo**
   ```python
   # Correcto (SQLAlchemy 2.0)
   name: Mapped[str] = mapped_column(String(100))
   
   # Incorrecto (versión antigua)
   name = Column(String(100))
   ```

### Error: "relation already exists"

**Causa:** La tabla existe en BD pero Alembic no lo sabe.

**Solución:**
```bash
# Marcar la BD como actualizada sin ejecutar cambios
alembic stamp head
```

### Error: "type XXX already exists" (ENUM en PostgreSQL)

**Causa:** Los tipos ENUM de PostgreSQL persisten independientemente de las tablas.

**Solución manual:**
```python
# En psql o con SQLAlchemy
drop type if exists coursetype cascade;
drop type if exists userrole cascade;
drop type if exists paymentstatus cascade;
drop type if exists classstatus cascade;
drop type if exists dayofweek cascade;
```

### Error: "can't proceed with" (dependencias circulares)

**Causa:** Cambios complejos que requieren pasos múltiples.

**Solución:** Crear migración manual o dividir en pasos:
```python
def upgrade() -> None:
    # Paso 1: Crear nueva columna
    op.add_column('table', sa.Column('new_col', sa.String(), nullable=True))
    
    # Paso 2: Migrar datos
    op.execute("UPDATE table SET new_col = old_col")
    
    # Paso 3: Hacer NOT NULL
    op.alter_column('table', 'new_col', nullable=False)
    
    # Paso 4: Eliminar columna antigua
    op.drop_column('table', 'old_col')
```

### Necesito modificar una migración ya creada (no aplicada)

```bash
# 1. Eliminar el archivo de migración
rm migrations/versions/xxxx_nombre.py

# 2. Recrear
alembic revision --autogenerate -m "nuevo nombre"
```

### Migración aplicada en desarrollo, necesito cambiarla

```bash
# 1. Revertir
alembic downgrade -1

# 2. Eliminar archivo
rm migrations/versions/xxxx_nombre.py

# 3. Recrear y aplicar
alembic revision --autogenerate -m "nombre corregido"
alembic upgrade head
```

## Advertencias de Seguridad

### ⚠️ NUNCA hagas esto en producción:

```python
# DESTRUCTIVO - Pierdes todos los datos
db.drop_all()
db.create_all()
```

```bash
# DESTRUCTIVO - Revertir todo puede causar pérdida de datos
alembic downgrade base
```

```python
# En una migración - Destruye datos existentes
op.execute("DELETE FROM table")
```

### ✅ Siempre en producción:

1. **Crear backup antes de cualquier migración**
   ```bash
   pg_dump -U postgres legion_combat_prod > backup.sql
   ```

2. **Probar migraciones en staging primero**

3. **Revisar el script antes de aplicar**
   ```bash
   alembic upgrade head --sql  # Ver SQL sin ejecutar
   ```

4. **Migraciones atómicas**: usar transacciones cuando sea posible
   (PostgreSQL ya maneja esto automáticamente)

5. **Migraciones reversibles**: siempre escribir downgrade()

### Buenas prácticas adicionales

1. **Un cambio lógico por migración**
   - No mezclar "agregar columna" con "crear tabla nueva"
   - Facilita debugging y rollback

2. **Nombres descriptivos**
   - ✅ `agregar_campo_email_a_usuarios`
   - ❌ `update`

3. **Revisar siempre el SQL generado**
   ```bash
   alembic upgrade head --sql > migration.sql
   cat migration.sql
   ```

4. **No modificar migraciones ya aplicadas**
   - Crear nueva migración si necesitas corregir algo

## Referencia Rápida

### Operaciones Alembic comunes

| Operación | Comando Alembic |
|-----------|-----------------|
| Crear tabla | `op.create_table()` |
| Eliminar tabla | `op.drop_table()` |
| Agregar columna | `op.add_column()` |
| Eliminar columna | `op.drop_column()` |
| Modificar columna | `op.alter_column()` |
| Crear índice | `op.create_index()` |
| Eliminar índice | `op.drop_index()` |
| Crear foreign key | `op.create_foreign_key()` |
| Ejecutar SQL | `op.execute()` |

### Tipos de datos SQLAlchemy

```python
sa.String(length=100)      # VARCHAR
sa.Text()                  # TEXT
sa.Integer()               # INTEGER
sa.Float()                 # FLOAT
sa.Numeric(10, 2)          # DECIMAL(10,2)
sa.Boolean()               # BOOLEAN
sa.Date()                  # DATE
sa.DateTime(timezone=True) # TIMESTAMP WITH TIME ZONE
sa.Uuid()                  # UUID
sa.Enum(EnumClass)         # ENUM
```
